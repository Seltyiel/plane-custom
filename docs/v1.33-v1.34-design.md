# 1. Permessi: cosa abbiamo gia' e cosa serve

Plane offre **2 livelli x 3 ruoli**:

- Livelli (`EUserPermissionsLevel`): `WORKSPACE` / `PROJECT`
- Ruoli (`ROLE` Python = `EUserPermissions` TS): `ADMIN=20` / `MEMBER=15` / `GUEST=5`

I due livelli sono indipendenti: si puo' essere `MEMBER` del workspace ma `ADMIN` di un singolo progetto. Tabelle DB: `WorkspaceMember.role`, `ProjectMember.role`. Decoratore backend: `@allow_permission([ROLE.ADMIN, ROLE.MEMBER], level="WORKSPACE")`. Hook frontend: `useUserPermissions().allowPermissions(...)`.

**Il modello attuale non basta per i Meeting privati**: "visibili solo agli invitati" non e' un concetto di ruolo, e' una ACL per-evento. Lo gestiamo con la tabella `MeetingAttendee` (vedi sotto).

Per il Time tracking i ruoli stock bastano:
- `MEMBER`: logga le proprie ore, le edita finche' non sono approvate.
- `ADMIN`: logga proprio + edita/cancella ore di altri + approva.
- `GUEST`: NON logga ore (read-only sul timesheet del workspace).

## 2.2 Schema DB

```sql
-- Tabella principale: ogni riga = una sessione di lavoro loggata.
CREATE TABLE time_logs (
  id            UUID PRIMARY KEY,
  workspace_id  UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  project_id    UUID NULL REFERENCES projects(id) ON DELETE CASCADE,    -- nullable se workspace-level task
  issue_id      UUID NULL REFERENCES issues(id) ON DELETE CASCADE,      -- nullable se "ore generiche"
  user_id       UUID NOT NULL REFERENCES users(id),

  duration_seconds INT NOT NULL CHECK (duration_seconds > 0 AND duration_seconds < 86400 * 7),
  logged_at     TIMESTAMP WITH TIME ZONE NOT NULL,        -- quando il lavoro e' stato fatto
  description   TEXT NULL,                                 -- nota libera

  -- Tracking provenienza
  source        VARCHAR(16) NOT NULL DEFAULT 'manual',    -- 'manual' | 'timer'
  timer_started_at  TIMESTAMP WITH TIME ZONE NULL,        -- popolato solo se source='timer'

  -- Approval workflow
  approval_status VARCHAR(16) NOT NULL DEFAULT 'pending', -- 'pending' | 'approved' | 'rejected' | 'auto'
  approved_by_id  UUID NULL REFERENCES users(id),
  approved_at     TIMESTAMP WITH TIME ZONE NULL,
  rejection_reason TEXT NULL,

  created_at    TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
  deleted_at    TIMESTAMP WITH TIME ZONE NULL              -- soft delete come Issue
);

CREATE INDEX time_logs_user_logged_at ON time_logs(user_id, logged_at DESC);
CREATE INDEX time_logs_issue ON time_logs(issue_id);
CREATE INDEX time_logs_workspace_period ON time_logs(workspace_id, logged_at);
CREATE INDEX time_logs_pending ON time_logs(workspace_id, approval_status) WHERE approval_status = 'pending';

-- Tabella separata per il timer attivo. Ogni utente ha al massimo 1 timer attivo.
CREATE TABLE active_timers (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,  -- UNIQUE → 1 sola sessione attiva per user
  workspace_id  UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  issue_id      UUID NULL REFERENCES issues(id) ON DELETE SET NULL,
  started_at    TIMESTAMP WITH TIME ZONE NOT NULL,
  description   TEXT NULL                                  -- l'utente puo' annotare durante
);
```

**Note di design:**

- `source='manual'`: l'utente compila un form "ho fatto X ore Y minuti il giorno Z"
- `source='timer'`: lo creiamo automaticamente allo Stop, salvando `started_at` originale e calcolando `duration_seconds = stop - start`
- `approval_status='auto'` (non pending): impostato automaticamente quando il workspace setting `time_tracking_approval_required=false`
- `issue_id` nullable per fare "Generic time" (es. "1h riunione interna senza task associato"). Ma in MVP **forziamo issue_id NOT NULL** per non aggiungere casi edge: tutti i log devono essere su un task. (Se vuoi cambiare scrivimi.)
- `deleted_at`: soft delete (pattern Plane). Permette di "ripristinare un log cancellato" senza perdere storia.

## 2.3 Settings

Estensione **Workspace** settings (no per-project: tenere globale e' meno overhead):

| Campo | Default | Descrizione |
|-------|---------|-------------|
| `time_tracking_enabled` | `false` | Master switch. Se OFF, niente UI ne' API. |
| `time_tracking_timer_enabled` | `true` | Se ON, mostra anche il pulsante timer; se OFF solo manual entry |
| `time_tracking_approval_required` | `false` | Se ON, i log devono essere approvati prima di entrare nei report aggregati |
| `time_tracking_billable_default` | `true` | Pre-flag billable per nuovi log (campo aggiungibile in v1.35 se serve) |

DB: ne' una nuova tabella ne' un campo su Workspace. Lo metto su una tabella `workspace_feature_settings` riusabile per future feature toggles. Schema:

```sql
CREATE TABLE workspace_feature_settings (
  id           UUID PRIMARY KEY,
  workspace_id UUID NOT NULL UNIQUE REFERENCES workspaces(id) ON DELETE CASCADE,
  -- JSON object con tutti i flag, espandibile
  features     JSONB NOT NULL DEFAULT '{}'::jsonb
);
```

Esempio valore: `{"time_tracking_enabled": true, "time_tracking_approval_required": false, "meetings_enabled": false}`. Lazy create alla prima query (come il workspace project v1.22).

## 3.3 Permessi (privacy "solo agli invitati")

**Visibilita'** del meeting:
1. Il `created_by` lo vede sempre.
2. Tutti gli `attendees` con `user_id` (interni) lo vedono.
3. Tutti gli altri **NON lo vedono**, neanche gli admin del workspace.
4. Eccezione opzionale (default OFF, per "audit trail"): un workspace setting `meetings_admin_visibility` = se ON, gli admin vedono solo titolo+orario+counter di tutti i meeting (non descrizione, non attendee list). Per ora **OFF di default**.

**Mutazioni**:
- Solo `created_by` puo' edit/cancel.
- Solo `created_by` puo' aggiungere/rimuovere attendees.
- Tutti gli attendees possono fare RSVP solo sul proprio status.
- Eccezione: se il `created_by` e' deactivato/rimosso dal workspace, l'admin **del workspace** puo' subentrare e modificare il meeting (ownership transfer).

**Implementazione**: queryset filter manuale in ogni endpoint, perche' non possiamo affidarci a `ProjectMember` come fa Plane di default. Pseudo-codice:

```python
def visible_meetings_for_user(user, workspace):
    return Meeting.objects.filter(workspace=workspace).filter(
        Q(created_by=user) |
        Q(meeting_attendees__user=user)
    ).distinct()
```

E permission class custom `MeetingVisibilityPermission` che lo applica.

## 3.5 Email & notifiche flow

### Invite (al create / add-attendee)
1. Backend crea `Meeting` + `MeetingAttendee` records.
2. Per ogni attendee, dispatch Celery task `send_meeting_invite(attendee_id)`:
   - Genera template HTML (riusa stile email Plane esistente).
   - Allega file ICS standard (RFC 5545) cosi' Outlook/Gmail offrono "Add to calendar".
   - Manda con `EmailMultiAlternatives`.
   - Aggiorna `invitation_email_sent_at`.

### Update (edit di start/end/location)
- Email "Meeting updated" con diff prima/dopo + nuovo ICS (METHOD=REQUEST).
- In-app notification per attendees registrati.

### Cancel
- Email cancellation (METHOD=CANCEL nel ICS).
- In-app notification.
- `Meeting.cancelled_at` settato. Soft delete.

### Reminder
Celery beat task scheduled ogni 1 minuto:
```python
@shared_task
def dispatch_meeting_reminders():
    now = timezone.now()
    cutoff = now + timedelta(minutes=15)
    qs = MeetingAttendee.objects.filter(
        meeting__start_at__gte=now,
        meeting__start_at__lte=cutoff,
        meeting__cancelled_at__isnull=True,
        meeting__deleted_at__isnull=True,
        reminder_inapp_sent_at__isnull=True,
        status__in=['accepted', 'tentative', 'invited'],   # non rimindiamo i declined
    )
    for attendee in qs:
        # in-app notification
        if attendee.user_id:
            create_inapp_notification(attendee.user_id, ...)
            attendee.reminder_inapp_sent_at = now
        # email se SMTP configurato
        if smtp_configured():
            send_meeting_reminder_email(attendee.id)
            attendee.reminder_email_sent_at = now
        attendee.save()
```

Beat schedule entry: `{ 'meeting-reminders': { 'task': 'plane.bgtasks.meeting.dispatch_meeting_reminders', 'schedule': 60.0 } }`.

## 3.6 Recurrence (v1.35)

Salvo l'rrule sul Meeting "master". Quando il frontend richiede meeting in [from, to]:

1. Backend prende tutti i master non-ricorrenti che si intersecano con [from, to].
2. Per ogni master con `recurrence_rule != null`, espande con `python-dateutil.rrule.rrulestr` su [from, to], skippando `excluded_dates` e ottenendo lista di date.
3. Per ogni data, crea un dict "istanza virtuale" con `id=<masterId>:<date>`.
4. Sostituisce con eventuali `child` Meeting esistenti per quella data (override).
5. Ritorna lista flat ordinata per `start_at`.

Frontend tratta le istanze virtuali come oggetti normali. Edit di una singola istanza chiama POST `/override-occurrence/` che crea un child Meeting "vero" (con `parent_meeting_id` puntato al master). Dopo, quel child si comporta indipendentemente.

**Limitazioni v1.35**: solo daily/weekly/monthly base + interval. Niente UI per RRULE custom complesse tipo "secondo lunedi' del mese".

# 4. Integrazione tra le due feature

Time tracking + Meetings interagiscono in 2 modi opzionali (v1.36+, fuori scope MVP):

1. **Auto-log da meeting**: alla fine di un meeting, l'attendee puo' "Logga il tempo passato qui sul task linkato" → preset duration = meeting duration, issue = primo linked issue.
2. **Time tracking del meeting**: aggiungere ore al meeting senza creare un TimeLog separato. Probabilmente NO - meglio tenere le 2 entita' indipendenti e usare il link.

In MVP non implementiamo nessuna di queste. Il link via `MeetingIssueLink` basta per navigazione manuale.

# 5. Rischi & open issues

| # | Rischio | Mitigazione |
|---|---------|-------------|
| R1 | SMTP non configurato in prod → email non parte → Meeting MVP useless | Documentare in CHANGELOG + UI: se SMTP non configurato, banner "Configure SMTP in god mode to enable meeting emails" |
| R2 | Recurrence e' complessa (v1.35), rischio di bug subdoli (DST, exception dates, parent vs child) | Iniziare con tests massicci sul backend RRULE expander prima di toccare UI. Limitare RRULE a daily/weekly/monthly base in v1.35. |
| R3 | "Privacy solo invitati" puo' leakare via search globale o sidebar count | Audit: ogni endpoint che restituisce meeting o conta meeting deve usare `visible_meetings_for_user`. Test E2E con utenti diversi. |
| R4 | Linking meeting a issue di progetti diversi (workspace meeting) richiede usare workspace project pattern (v1.22) | Riusiamo il pattern workspace-hidden project per "linkare a un task qualsiasi nel workspace anche se il meeting e' workspace-level". |
| R5 | Time tracking timer perso al refresh / navigazione | Lo stato e' in DB (ActiveTimer table), non in browser. Il banner SWR-poll periodico ricostruisce. |
| R6 | Migration su DB grande puo' essere lenta (millioni di issues per indice time_logs) | Indici creati con `CONCURRENTLY` se necessario, in produzione. |
| R7 | Calendar layout overlay tasks+meetings → race condition fetch (v1.16 simile) | Una sola fetch combinata per il calendar layout, response shape `{tasks: [...], meetings: [...]}`. Riusiamo la lezione v1.16. |
